# 金字塔加碼滾動式策略 — 修訂版規格（市場狀態機）

> 本文件以顧問提供的原始設計為基礎，補齊實作前必須訂死的規則缺口。
> 缺口修訂處以「**修訂**」標註，並附理由。實作時以本文件為準。
>
> 狀態：規格定稿，尚未實作。實作時歸屬 `src/strategy-engine/`，
> 依專案規則不得 import 任何 RN/Expo 套件，必須可獨立跑 Jest 測試。

---

## 1. 核心思路

用「市場狀態機」取代固定百分比觸發：每個 bar（日線）重新判斷市場狀態，
加碼邏輯依當下結構動態切換規則，而不是一套公式套用全程。

- 加碼只能在 `TRENDING_UP` 或 `BREAKOUT_UP` 狀態觸發
- `CONSOLIDATION` 一律凍結加碼並收緊移動停損
- `TRENDING_DOWN` / `BREAKOUT_DOWN` 停止加碼並檢查停損出場

---

## 2. 狀態定義與判斷規則

### 2.1 狀態列表

| 狀態 | 判斷依據 |
|---|---|
| `TRENDING_UP` | 短均線 > 長均線（如 20MA > 60MA）且收盤價站上短均線 |
| `TRENDING_DOWN` | 短均線 < 長均線且收盤價跌破短均線 |
| `CONSOLIDATION` | 見 2.2 的多數決規則 |
| `BREAKOUT_UP` / `BREAKOUT_DOWN` | 前一狀態為 `CONSOLIDATION`，收盤價站上/跌破整理區間邊界，且量能達均量 `volume_confirm_ratio` 倍以上 |

### 2.2 盤整判斷 —— 多數決規則（修訂 #1）

原設計「三者任一成立」未訂優先權，且單一指標易誤判。**修訂為：以下三項中
「任兩項成立」即判定為盤整候選**：

1. 均線糾結：|20MA − 60MA| / 60MA < `ma_converge_pct`（預設 2%）
2. 區間振幅收斂：近 `consolidation_lookback` 日 (最高 − 最低) / 最低 < `range_narrow_pct`（預設 6~8%）
3. ATR 收縮：近期 ATR(14) < 前一期 ATR(14) × `atr_shrink_ratio`（預設 0.8）

> 理由：任一成立太鬆（單指標誤判率高），三者全成立太嚴（幾乎不會觸發）。
> 兩項成立是明確、無隱藏優先權的規則，行為由設計決定而非程式碼順序決定。

### 2.3 狀態切換需確認天數（修訂 #3）

原設計每根 bar 獨立判斷，指標在門檻邊緣時狀態會每日抖動（TRENDING ↔
CONSOLIDATION 閃爍），加碼凍結/解凍跟著閃爍。**修訂為：**

- 候選狀態需**連續 `state_confirm_days` 天（預設 2）判斷結果一致**，才正式切換狀態
- 未滿確認天數前，維持前一個已確認狀態
- **例外：`BREAKOUT_UP` / `BREAKOUT_DOWN` 不需確認天數**，因為突破本身已用
  量能過濾假訊號，再等確認天數會錯過突破意義

> 此設計比照現有 `trend-classifier.ts` 的 `confirmDays` 概念。

### 2.4 BREAKOUT 是一次性事件（修訂 #4）

原設計未定義 BREAKOUT 之後如何轉換。**修訂為明確規則：**

- `BREAKOUT_UP` / `BREAKOUT_DOWN` 只存在**一根 bar**，屬於「事件」而非「持續狀態」
- 突破當根 bar 執行對應行為（加碼 / 停損檢查）後，**隔天起交回 2.1 的一般
  狀態判斷**（自然流向 `TRENDING_UP`、`TRENDING_DOWN`，或被打回 `CONSOLIDATION`）
- 狀態歷史記錄中 BREAKOUT 照常記一筆，方便回測時對齊事件

### 2.5 量能確認逾時（修訂 #6）

價格突破區間邊界但量能未達 `volume_confirm_ratio` 時：

- 進入「待確認」計數，最多等 `breakout_confirm_days` 天（預設 3）
- 期限內任一天量能補上且價格仍在區間外 → 當天觸發 BREAKOUT
- 逾期未確認 → 視為假突破，打回 `CONSOLIDATION`，計數歸零

## 3. 各狀態對應的加碼與停損行為

| 狀態 | 加碼 | 停損 |
|---|---|---|
| `TRENDING_UP` | 允許：現價 ≥ 上一次加碼價 × (1 + `add_trigger_pct`) 時觸發下一級 | **趨勢中也要移動停損（修訂 #2，見 3.1）** |
| `CONSOLIDATION` | 凍結，不出場 | 停損上移至 max(現有停損, 最近一次加碼價 × (1 − `stop_buffer_pct`)) |
| `BREAKOUT_UP` | 觸發下一級加碼（盤整後唯一的加碼入口） | 依 `TRENDING_UP` 規則接手 |
| `BREAKOUT_DOWN` / `TRENDING_DOWN` | 停止加碼 | 檢查是否跌破移動停損 → 觸發即出場或減碼 |

### 3.1 TRENDING_UP 的移動停損（修訂 #2）

原設計只在進入盤整時收緊停損。若股價在趨勢中一路上漲、遲遲不進入盤整，
停損會停留在很早期的加碼價，帳面獲利完全沒鎖住。**修訂為：**

- `TRENDING_UP` 期間，每根 bar 更新：停損 = max(現有停損, 短均線 × (1 − `trail_ma_buffer_pct`))
- 停損**只上移不下移**（ratchet，棘輪式）
- 盤整時的「加碼價 × (1 − stop_buffer_pct)」規則與此取 max，兩者不衝突

### 3.2 加碼觸發基準：相對上一級加碼價

觸發條件用「相對前一次加碼價的漲幅」而非「相對起始成本」，避免趨勢後段
間距過鬆或過緊。金字塔權重沿用 1 : 1.5 : 2 : 2.5 : 3 ...（與現有
`grid-strategy.ts` 的 `tierWeight` 一致）。

> **架構影響（修訂 #5）**：這不是加一個參數就好。現有 `grid-strategy.ts`
> 是「固定錨點算靜態檔位」的無狀態純函式；本策略需要記錄
> 「已加碼到第幾級、上一次加碼價、目前停損價、目前已確認狀態、
> 待確認計數」等**持久化狀態**。實作時：
>
> - 策略函式本身仍保持純函式：`(history, config, prevState) → (signal, nextState)`，
>   狀態進、狀態出，不在函式內做 I/O，維持可獨立測試
> - 狀態的持久化由呼叫端（DB repo 層）負責，資料模型需新增一張表或欄位
>   儲存 `PyramidState`（見 §5）
> - 這是與現有三個策略最大的差異，工時估算要把 DB schema 變更與
>   遷移算進去

## 4. 參數配置（全部外部化）

```
entry_price             # 起始成本
budget                  # 總預算（涵蓋起始部位；各級金額 = budget × weight / Σweights）
weights                 # [1, 1.5, 2, 2.5, 3, ...] 金字塔權重，weights[0] 為起始部位
ma_short / ma_long      # 趨勢判斷均線週期（預設 20 / 60）
atr_period              # ATR 週期（預設 14，外部化以利測試與回測調整）
ma_converge_pct         # 均線糾結門檻（預設 2%）
consolidation_lookback  # 區間高低點回看天數（預設 20）
range_narrow_pct        # 區間振幅盤整門檻（預設 6~8%）
atr_shrink_ratio        # ATR 收縮比率門檻（預設 0.8）
state_confirm_days      # 狀態切換確認天數（預設 2；BREAKOUT 豁免）
volume_confirm_ratio    # 突破量能需達均量倍數（預設 1.2~1.5）
breakout_confirm_days   # 量能確認逾時天數（預設 3）
stop_buffer_pct         # 盤整時移動停損緩衝（預設 3~5%）
trail_ma_buffer_pct     # 趨勢中跟隨短均線的停損緩衝（預設 2~3%）
add_trigger_pct         # 相對上一級加碼價的觸發漲幅（預設 5%）
```

## 5. 需要持久化的狀態（PyramidState）

```
current_state        # 目前已確認狀態
candidate_state      # 候選狀態（確認天數計數中）
candidate_days       # 候選狀態已連續天數
current_tier         # 已加碼到第幾級（0 = 只有起始部位）
last_add_price       # 上一次加碼價（第 0 級時 = entry_price）
stop_price           # 目前移動停損價
breakout_pending_days # 量能待確認計數（0 = 無待確認突破）
range_high / range_low # 進入盤整時鎖定的區間邊界
```

每次狀態變化寫入歷史記錄（時間、前狀態、後狀態、觸發原因），供回測與除錯。

## 6. Trade-off 總覽

| 設計選擇 | 優點 | 代價 |
|---|---|---|
| 盤整判斷採「三取二」多數決 | 無隱藏優先權、降低單指標誤判 | 比「任一成立」晚確認盤整 |
| 狀態切換需確認天數 | 消除門檻邊緣的狀態抖動 | 狀態轉換慢 1~2 天，突破豁免以免錯過 |
| 突破需量能確認 + 逾時規則 | 過濾假突破且不會無限期等待 | 真突破但量能延後放大時，最多晚 `breakout_confirm_days` 天觸發 |
| 加碼觸發相對上一級 | 趨勢後段間距不失真 | 策略從無狀態變有狀態，需 DB 持久化，架構工時增加 |
| 趨勢中也移動停損（跟短均線） | 趨勢段帳面獲利有鎖住 | 急漲後貼均線整理可能被提早停損掃出 |
| 盤整只凍結不出場 | 避免真突破前被洗出場 | 盤整演變成反轉時，出場比直接出場的做法晚 |

## 7. 實作者需求一句話（更新版）

> 建一個狀態機，每天用（股價、均線、ATR、成交量）判斷市場狀態，
> 狀態切換需連續 N 天確認（突破豁免）；盤整用三指標取二的多數決；
> 金字塔加碼只在 `TRENDING_UP` / `BREAKOUT_UP` 觸發，觸發基準為相對
> 上一次加碼價；`CONSOLIDATION` 凍結加碼並上移停損；`TRENDING_UP` 期間
> 停損棘輪式跟隨短均線；突破需量能確認且有逾時打回機制；
> 策略函式保持 `(history, config, prevState) → (signal, nextState)` 純函式，
> 狀態持久化交給呼叫端；所有閾值參數外部化供回測調整。

## 8. UI 輸出原則：讓使用者「按表操課」

產品目標是讓使用者依資訊與策略穩定累積資產，因此策略輸出到 UI 時
**必須是可直接執行的行動指示，不是需要使用者再解讀的技術指標**。

每檔股票每天呈現一張「今日操作卡」，內容固定四欄：

1. **今日動作**：加碼（含金額與第幾級）／凍結觀望／續抱／出場減碼，
   直接對應 `PyramidSignal.action`，用大字與顏色區分（加碼綠、出場紅、
   凍結灰、續抱藍）
2. **停損價**：目前移動停損價，跌破就出場——這是使用者唯一需要記住的數字
3. **原因**：一句話說明為什麼（直接用 `signal.reason`），讓使用者理解但
   不需要理解也能操作
4. **狀態**：目前市場狀態（趨勢/盤整/突破）用圖示呈現，僅供參考

原則：使用者不需要看懂均線、ATR、量能比——那些是引擎內部判斷依據；
UI 只要回答「我今天要做什麼、金額多少、防守價在哪」三個問題。

## 附錄 A：籌碼濾網（第二階段擴充，狀態機本體完成並回測後再做）

### A.1 定位原則

籌碼資料**不新增狀態**，狀態機維持四態不動。籌碼只作為加碼動作前的
最後一道確認濾網（角色類似現有 `entry-advisor.ts` 中 trend-classifier
的安全閥門）：狀態機說「可以加碼」之後，濾網有權把動作**降級或凍結**，
但無權主動觸發加碼。

濾網獨立成模組（如 `chip-filter.ts`），可獨立測試、獨立開關
（config 中 `chip_filter_enabled`），關閉時策略行為與基礎版完全一致。

### A.2 量價背離檢查（不需新資料源）

`TRENDING_UP` 觸發加碼時，檢查近 `divergence_lookback` 日（預設 10）：
若價創區間新高但同期均量低於前一期均量 × `volume_divergence_ratio`
（預設 0.7），視為量價背離，該級加碼金額乘以 `divergence_scale`
（預設 0.5）或跳過（可設定）。

### A.3 法人快濾網（每日資料）

- 資料源：TWSE 公開資料「三大法人買賣超（個股）」，每日更新，免費
- 規則：連續 `inst_sell_days`（預設 3）日法人合計賣超 → 加碼凍結，
  直到出現任一日買超才解除
- 只影響加碼，不影響停損與出場邏輯

### A.4 大戶慢濾網（每週資料）

- 資料源：TDCC 開放資料「集保戶股權分散表」，每週更新，免費
- 規則：持股 `major_holder_level`（預設 400 張以上級距）比例
  連續 `major_holder_weeks`（預設 2）週下降 → 整體轉保守：
  後續每級加碼金額乘以 `conservative_scale`（預設 0.5）
- **回測注意**：週資料公布有延遲，回測時只能使用「當時已公布」的
  那一期資料，避免 look-ahead bias

### A.5 新增參數

```
chip_filter_enabled      # 濾網總開關（預設 false，基礎版行為不變）
divergence_lookback      # 量價背離回看天數（預設 10）
volume_divergence_ratio  # 均量萎縮比率門檻（預設 0.7）
divergence_scale         # 背離時加碼金額縮放（預設 0.5，0 = 跳過）
inst_sell_days           # 法人連續賣超凍結天數（預設 3）
major_holder_level       # 大戶級距門檻（預設 400 張）
major_holder_weeks       # 大戶比例連降週數（預設 2）
conservative_scale       # 保守模式加碼縮放（預設 0.5）
```

### A.6 基礎建設需求

- 新增 `data-fetch/` client 抓取法人買賣超與集保資料（沿用現有
  重試與格式驗證慣例）
- 新增 DB 表儲存籌碼歷史（法人日資料、集保週資料）
- 濾網模組同樣不得 import RN/Expo 套件，輸入為已讀出的籌碼資料陣列

### A.7 驗收標準

先完成狀態機本體回測，再開啟濾網跑同一段歷史資料比較：
**濾網必須在回測中證明有改善（報酬或最大回撤至少一項變好、另一項
不顯著變差）才保留**，否則移除或維持預設關閉。

### A.8 明確排除項：市場消息

新聞/消息面**不進策略引擎**，理由：無免費結構化資料源、情緒判讀無法
回測驗證、且違背純函式可測試架構。消息面分析交由 Phase 6 的
Shortcuts 匯出流程：使用者手動觸發，把 App 匯出的策略數據交給
Claude App 結合當下消息給定性意見。
